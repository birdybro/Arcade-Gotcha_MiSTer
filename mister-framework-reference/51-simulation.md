## 1. Purpose & one-line summary

The MiSTer framework provides no built-in RTL testbench: `Template_MiSTer` ships no `tb/`, `sim/`, or `*_tb.sv` files and no simulation makefile. This document describes the conventions cores use when bringing up a Verilator or ModelSim/Questa testbench around `emu` — what must be stubbed, how to load ROMs, how to capture video/audio frames — and acknowledges that the only in-archive prose docs near this topic (`analogsim.md`, `debugging.md`) cover adjacent concerns (SPICE/Falstad analog modeling and JTAG hardware debug), not RTL TB scaffolding.

## 2. The contract (must-obey)

The framework does not define a simulation contract; the following are conventions and inferences a TB author must respect to bring `emu` up under simulation.

- The `Template_MiSTer` repository ships no testbench files of any kind under `Template_MiSTer/` or `Template_MiSTer/sys/`. [O] (archive/github/MiSTer-devel/Template_MiSTer/Readme.md:8-28 @ f35083f3b40d)
- The "Source structure" section of `Template_MiSTer/Readme.md` enumerates `sys`, `rtl`, `releases`, plus `.qpf/.qsf/.sdc/.srf/.sv/.qip/clean.bat/.gitignore/jtag.cdf` — it does not list any testbench, simulation script, or waveform configuration file. [O] (archive/github/MiSTer-devel/Template_MiSTer/Readme.md:13-27 @ f35083f3b40d)
- The in-archive prose doc `analogsim.md` describes SPICE/Falstad analog circuit modeling and HDL re-implementation of audio circuits, not RTL testbench simulation; it does not prescribe Verilator, ModelSim, `$readmemh`, clock models, or `emu`-level testbenches. [O] (archive/github/MiSTer-devel/MkDocs_MiSTer/docs/developer/analogsim.md:1-130 @ 9033bd292fdc)
- The in-archive prose doc `debugging.md` covers USB Blaster / JTAG upload from Quartus to MiSTer hardware; it does not describe RTL simulation. [O] (archive/github/MiSTer-devel/MkDocs_MiSTer/docs/developer/debugging.md:1-15 @ 9033bd292fdc)
- The only in-archive pointer to an RTL simulation flow is a link to "JimmyStones' Verilator Template" in `links.md`, which is an external repository not fetched into this archive. [O] (archive/github/MiSTer-devel/MkDocs_MiSTer/docs/developer/links.md:13 @ 9033bd292fdc)
- A TB instantiates the `emu` module verbatim from `Template.sv`; the port list comes from `sys/emu_ports.vh` included at the module header. [V] (archive/github/MiSTer-devel/Template_MiSTer/Template.sv:19-22 @ f35083f3b40d)
- `clk_sys` is the single clock domain on which `emu` runs, derived from the core's PLL in synthesis; in a TB it is generated by a behavioural `always #N clk_sys = ~clk_sys;` loop. [I] (archive/github/MiSTer-devel/Template_MiSTer/Template.sv:94-99 @ f35083f3b40d)
- Reset is driven by the framework in synthesis; under simulation the TB asserts the core's internal reset for a defined number of cycles, then deasserts. The framework provides no canonical TB reset sequence. [I]
- ROM payloads delivered through `hps_io`'s `ioctl_*` path in hardware must be supplied to the core via a TB-side stimulus in simulation; the canonical Verilog mechanism is `$readmemh`/`$readmemb` into a memory array, then driving `ioctl_download`/`ioctl_wr`/`ioctl_addr`/`ioctl_dout` from a stimulus block. [I]
- `sys/` files that instantiate Altera megafunctions (Cyclone V SoC HPS bridges, PLLs, on-chip RAM macros, SDRAM controller) cannot be simulated directly by a vendor-neutral simulator without IP simulation libraries; cores either stub these modules or simulate `emu` standalone with the `sys/` framework removed. [I] (archive/github/MiSTer-devel/Template_MiSTer/sys/hps_io.sv, sd_card.sv, sysmem.sv, sys_top.v @ f35083f3b40d)
- The HDMI scaler `ascal.vhd` is a large VHDL module (≈thousands of lines) that consumes most simulation time when included; cores typically bypass it in TB and observe analog/direct video output before scaling. [I] (archive/github/MiSTer-devel/Template_MiSTer/sys/ascal.vhd:1-30 @ 9033bd292fdc)
- SystemVerilog-only constructs (interfaces, packed structs, `always_comb`, `logic`) appear in several `sys/` files; Verilator supports these but with version-dependent caveats — pre-4.x Verilator lacked complete SV interface support. [I] (archive/github/MiSTer-devel/Template_MiSTer/sys/f2sdram_safe_terminator.sv, sys/yc_out.sv @ f35083f3b40d)
- Mixed-language (VHDL + Verilog) simulation requires a simulator that supports both languages — ModelSim/Questa do, free Verilator does not (Verilator is Verilog/SystemVerilog only). [I]
- Frame capture is performed by sampling `VGA_R/G/B/HS/VS/DE` (or equivalent analog/direct video outputs from `emu`) into a PPM/PNG file when `DE` is high and emitting a new file at each `VS` edge. The framework provides no helper for this — cores write their own. [I]
- Audio capture is performed by sampling `AUDIO_L/AUDIO_R` (16-bit signed) at the core's audio rate gated by its audio clock-enable, and emitting a `.wav` file at TB teardown. The framework provides no helper for this. [I]

## 3. Ports / signals reference

The TB instantiates `emu` with the port set from `sys/emu_ports.vh` (not enumerated here — see `10-emu-top-level.md`). The relevant TB-side concern is which `sys/` modules must be stubbed, replaced, or left intact when assembling the simulation. The table below classifies every file under `Template_MiSTer/sys/` by its simulation strategy.

```verilog

module emu
(
	`include "sys/emu_ports.vh"
);
```

## 4. Sequencing & timing

A TB lifecycle has six phases. None are dictated by the framework; the sequence below is the conventional shape inferred from how `emu` consumes its signals.

```
phase:   |  init  | reset | rom-load | run        | capture     | teardown
clk_sys  |________|‾|_|‾|_|‾|_|‾|_|‾|_|‾|_|‾|_|‾|_|‾|_|‾|_|‾|_|‾|________
reset_n  |________|_______/‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾‾________
ioctl_dl |________|_______/‾‾‾‾‾‾‾‾‾‾‾‾‾\__________________________________
ioctl_wr |________|_______|‾|_|‾|_|‾|_|‾|__________________________________
VS       |________________________________|‾|_____________|‾|______________
                                          ^ frame N done   ^ frame N+1 done
```

1. **init** — Open dump file (`$dumpfile`/`$dumpvars` or `VCD trace` in Verilator); zero all TB-driven inputs (`status = 0`, `ps2_key = 0`, `ioctl_* = 0`).
2. **reset** — Generate `clk_sys` (e.g. 50 MHz → `#10` period in `1ns` timescale); hold the core's internal reset (typically `RESET` from `emu_ports.vh` or a status-bit reset asserted via `status[0]`) for ≥ N cycles. The exact reset width is core-dependent and not framework-defined.
3. **rom-load** — Drive `ioctl_download = 1` and pulse `ioctl_wr` with `{ioctl_addr, ioctl_dout}` for each byte/word; deassert `ioctl_download` at end. Source bytes are loaded from a file via `$readmemh` into a TB array.
4. **run** — Free-run `clk_sys` for K cycles; the core advances and emits video/audio.
5. **capture** — On every `VS` edge, write the accumulated `(R,G,B,DE)` raster to a PPM/PNG file. On every audio CE pulse, append `(AUDIO_L, AUDIO_R)` 16-bit samples to a `.wav` buffer.
6. **teardown** — Close dump file; flush `.wav` header; `$finish`.

No phase is framework-prescribed; the only invariants are that `clk_sys` must toggle, the core's reset must release after PLL "lock" (which the TB fakes), and `ioctl_download` must be high during the entire ROM transfer.

## 5. Minimal working pattern

The framework provides no canonical testbench file. The skeleton below is **synthesized** — there is no upstream `tb.sv` to quote verbatim. The `emu` instantiation header is grounded in `Template.sv:19-23 @ f35083f3b40d`; the rest is conventional Verilator/iverilog scaffolding consistent with the practices implied by §4. Authority for the underlying flow is the `links.md` pointer to JimmyStones' Verilator Template (external, not fetched), referenced at `MkDocs_MiSTer/docs/developer/links.md:13 @ 9033bd292fdc`.

```verilog
// Synthesized example — no upstream source; emu signature per
// archive/github/MiSTer-devel/Template_MiSTer/Template.sv:19-23 @ f35083f3b40d
`timescale 1ns/1ps

module tb_emu;

  // ---- 1. clock model (20 ns period = 50 MHz) ----
  reg clk_sys = 0;
  always #10 clk_sys = ~clk_sys;

  // ---- 2. status word: status[0] is core reset per Template CONF_STR
  //         "T[0],Reset;" — see Template.sv:81-82 @ f35083f3b40d ----
  reg  [127:0] status      = 128'b0;
  reg  [10:0]  ps2_key     = 11'b0;

  // ---- 3. ROM/ioctl path stimulus ----
  reg          ioctl_download = 0;
  reg          ioctl_wr       = 0;
  reg  [26:0]  ioctl_addr     = 0;
  reg  [15:0]  ioctl_dout     = 0;

  reg  [7:0]   rom_image [0:65535];  // 64 KB ROM buffer

  // ---- 4. DUT — only the subset of emu ports the core actually drives
  //         is shown; stub the rest at module boundary (see §3 table) ----
  wire [7:0]   VGA_R, VGA_G, VGA_B;
  wire         VGA_HS, VGA_VS, VGA_DE;
  wire [15:0]  AUDIO_L, AUDIO_R;

  emu DUT (
    .CLK_50M     (clk_sys),       // see Template.sv emu_ports.vh
    .RESET       (status[0]),
    .HPS_BUS     (49'b0),         // hps_io stubbed; status/ioctl driven via TB hooks
    .VGA_R       (VGA_R), .VGA_G(VGA_G), .VGA_B(VGA_B),
    .VGA_HS      (VGA_HS), .VGA_VS(VGA_VS), .VGA_DE(VGA_DE),
    .AUDIO_L     (AUDIO_L), .AUDIO_R(AUDIO_R)
    // ... remaining emu ports tied / left unconnected per §3
  );

  // ---- 5. lifecycle ----
  initial begin
    $dumpfile("tb_emu.vcd");
    $dumpvars(0, tb_emu);

    // 5a. load ROM file
    $readmemh("rom_image.hex", rom_image);

    // 5b. release reset after 100 cycles
    status[0] = 1'b1;
    repeat (100) @(posedge clk_sys);
    status[0] = 1'b0;

    // 5c. deliver ROM via ioctl
    ioctl_download = 1'b1;
    for (int i = 0; i < 65536; i = i + 1) begin
      @(posedge clk_sys);
      ioctl_addr = i[26:0];
      ioctl_dout = {8'b0, rom_image[i]};
      ioctl_wr   = 1'b1;
      @(posedge clk_sys);
      ioctl_wr   = 1'b0;
    end
    ioctl_download = 1'b0;

    // 5d. free-run for two frames worth of cycles, then finish
    repeat (5_000_000) @(posedge clk_sys);
    $finish;
  end

  // ---- 6. frame capture on every VS ----
  always @(posedge VGA_VS) begin
    // write accumulated VGA_DE raster to PPM (omitted for brevity)
  end

endmodule
```

Notes on this skeleton:
- The signal `RESET` and the `HPS_BUS` bit-width come from `sys/emu_ports.vh`; their exact names and widths are documented in `10-emu-top-level.md` and are not re-derived here. [I]
- `status[0]` is wired to the OSD "Reset" button per the `Template_MiSTer` CONF_STR. [O] (archive/github/MiSTer-devel/Template_MiSTer/Template.sv:81-82 @ f35083f3b40d)
- Setting `HPS_BUS = 49'b0` is shorthand for "`hps_io` is being stubbed out at TB level"; in practice the TB either drives the `hps_io` outputs directly with `force`/`release` or replaces `hps_io` with a TB-friendly module exposing `status`/`ioctl_*`/`ps2_key` as inputs. [I]
- The audio-capture and frame-capture blocks are sketched, not fleshed out — actual PPM/WAV writing is conventional Verilog `$fwrite` against `$fopen`d handles. [I]

## 7. Anti-patterns

### A.1 Simulating `ascal.vhd` (full HDMI scaler) in the testbench

- **Symptom:** Simulation throughput collapses to a few hundred Hz of `clk_sys`; a single frame takes minutes to hours.
- **Cause:** `ascal.vhd` is a large adaptive scaler with internal frame buffers and polyphase filters; it executes orders of magnitude more events per `clk_sys` tick than `emu` itself, and is VHDL (incompatible with free Verilator).
- **Fix:** Drop `ascal.vhd` from the TB build. Observe video at the pre-scaler boundary (`VGA_R/G/B/HS/VS/DE` from `emu`, or `video_mixer.sv`'s output). Frame-compare at that interface, not at HDMI output.
- **Citation:** `archive/github/MiSTer-devel/Template_MiSTer/sys/ascal.vhd:1-30 @ 9033bd292fdc` (file header; full module is thousands of lines).

### A.2 Treating PLLs as instantaneous (zero-cycle lock)

- **Symptom:** Reset releases before the core's internal clocks would have stabilised in hardware; downstream synchronous logic exhibits TB-only behaviour that disappears in the bitstream.
- **Cause:** `sys/pll_*.v` are Altera megafunction wrappers; in simulation they are replaced by a behavioural clock generator, and the `locked` output is tied high immediately. Reset deassertion gated on `locked` then fires earlier in TB than in hardware.
- **Fix:** Either drive `clk_sys` with a small but realistic startup delay before deasserting reset (≥ 1 µs of simulated time), or hold reset for a fixed cycle count that mirrors the hardware sequence. Do not gate TB reset on a stubbed PLL `locked`.
- **Citation:** `archive/github/MiSTer-devel/Template_MiSTer/sys/pll_audio.v`, `sys/pll_hdmi.v` (file presence, observed @ f35083f3b40d) — these are megafunction wrappers, not behavioural models.

### A.3 Modeling the SDRAM controller as zero-latency RAM

- **Symptom:** Core works under TB, fails on hardware with corrupted ROM reads or scrambled video tiles; sometimes only at specific PLL frequencies.
- **Cause:** The real SDRAM controller has multi-cycle CAS latency, refresh cycles, and arbitration between multiple ports. A behavioural stub that returns data combinatorially on `addr` hides timing assumptions in the core.
- **Fix:** Use a behavioural SDRAM model that mimics CAS latency (typically 2–3 cycles) and busy/ready handshake. At minimum, return data on the second cycle after `req`. Where possible, simulate against the actual `sdram.v` from the core's `rtl/`.
- **Citation:** No direct citation in `Template_MiSTer` (the template lacks a core-side SDRAM controller). Inference grounded in the existence of SDRAM in the framework — see `archive/github/MiSTer-devel/Template_MiSTer/Template.sv:30 @ f35083f3b40d` (SDRAM pin set being tied off in the template).

### A.4 Trusting Verilator to compile every `sys/` file

- **Symptom:** Verilator emits `error: Unsupported: SystemVerilog 2005/...` on `sys/f2sdram_safe_terminator.sv`, `sys/yc_out.sv`, or VHDL `sys/ascal.vhd` / `sys/pll_hdmi_adj.vhd`.
- **Cause:** Verilator is Verilog/SystemVerilog only and has version-dependent gaps in SV interface, packed-struct, and `always_comb` corner cases. VHDL files cannot be processed at all.
- **Fix:** Build the TB only against the modules the test exercises (per the §3 table). Drop all VHDL. Pin a recent Verilator (≥ 4.2x) for SV interface support. For mixed-language sims, use ModelSim/Questa; for pure-SV cores, Verilator is preferred for speed.
- **Citation:** File listing of `archive/github/MiSTer-devel/Template_MiSTer/sys/` @ f35083f3b40d shows `.v`, `.sv`, and `.vhd` mixed.

## 8. Verification

The framework provides no test infrastructure. Verification of a TB-developed core takes one of these forms:

- **Waveform inspection.** Open `tb_emu.vcd` (or `.fst` from Verilator) in GTKWave; verify `clk_sys` toggle, `RESET` release, `ioctl_*` activity during ROM load, periodic `VS`. This is the universal smoke-test step.
- **Frame-level golden compare.** After each `VS`, write the captured raster (`VGA_R/G/B` masked by `VGA_DE`) to a PPM file. Compare PPM-by-PPM to a reference set produced from a known-good build. PPM files diff cleanly with `cmp`.
- **Audio WAV golden compare.** Write captured `AUDIO_L/AUDIO_R` to a `.wav` file at a chosen sample rate (gated by the core's audio CE). Compare to a reference WAV. For analog-circuit cores, `analogsim.md` describes the SPICE→WAV reference flow used to validate audio HDL. [O] (archive/github/MiSTer-devel/MkDocs_MiSTer/docs/developer/analogsim.md:85-103 @ 9033bd292fdc)
- **MiSTer.ini surface symptoms (post-bitstream).** Bugs that survive TB but appear on hardware usually map to: SDRAM latency mismatch (corrupted tiles → revisit A.3), PLL/clock-domain crossing (intermittent garble → revisit A.2), HDMI scaler interaction (visible only post-`ascal` → out of TB scope, see A.1).
- **JTAG / SignalTap on hardware.** When TB passes but hardware fails, capture on-chip with SignalTap. The connection details (USB Blaster on Linux) are in `debugging.md`. [O] (archive/github/MiSTer-devel/MkDocs_MiSTer/docs/developer/debugging.md:1-15 @ 9033bd292fdc)
